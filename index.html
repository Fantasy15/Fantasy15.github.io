  • 安装成功后

前端静态资源版本号的设置

发表于 2019-03-20 | 更新于: 2019-03-20 | | 阅读次数:

版本控制

用户对网络资源访问的时候，会对资源进行本地缓存，以便后续访问的时候能够直接从本地读取，加快访问速度，缓存的资源主要包括js, css, 图片等资源。

项目开发迭代的时候，需要对这些资源进行更新，就要加入版本控制。

现行的主要有两种缓存方式:

  • 加参数(覆盖式更新)
  • 改变文件名(非覆盖式更新)

以 js 举例就是 .js?v=xxxx 和 .xxxx.js, 浏览器会认为这两种都是新的链接，会发起新的资源请求。

哪种好；先说结论： .xxxx.js 是完美的版本控制方式, 但 .js?v=xxx 更普遍.

为什么呢，看下面的具体分析

覆盖式更新

  • 先部署页面，后部署资源。部署资源之前，用户访问页面，会请求旧的资源并缓存。除非手动刷新，否则不会请求新的资源。假若页面结构有变动，会引起样式错乱，执行错误。
  • 先部署资源，后部署页面。部署页面之前，用户访问页面，会出现旧页面访问新资源，假若页面有变动，可能会引起样式错乱，执行错误。

非覆盖式更新

  • 先部署页面，后部署资源。部署资源之前，用户访问页面，资源404。可能会样式错乱和空白。
  • 先部署资源，后部署页面。部署页面之前，用户访问页面，旧页面访问旧资源，后续新页面访问新资源，不会出现问题。

理论上 .xxxx.js 非覆盖式更新是完美的。 先部署资源，后部署页面。并删除老的资源。但因为 vue, react 兴起，现在很多页面都是只有一个 <div id="app"></div>，所以即使覆盖式更新哪种情况用户都不会出现样式错乱，执行错误问题。还要考虑现在很多的上线流程除非分开任务上，否则保证不了上线的顺序。

event loop

js 是一个单线程语言，event loop 是它的一个运行机制。在运行代码时，js 会将碰到的 异步任务 挂起，放到任务队列，继续执行下面的同步代码，并将同步代码执行完毕，在这个时候，任务队列中的任务开始执行，执行完之后会通知js，并执行后续的 callback.

简单的例子

demo1.js

代码顺序执行，

  • 输出 1。
  • 碰到 异步任务(setTimeout)，放到任务队列，即使 timeout 的参数是0， 然后输出3。
  • 执行任务队列中的 setTimeout ，输出2.

哪些任务会放到消息队列

异步任务; 包括 setTimeout, setInterval, ajax 都会触发 event loop 机制，被放到任务队列

移动端 web 调试存在调试工具不足，问题定位困难，环境配置复杂等问题，通过 chrome 浏览器的 inpect 功能 能够有效地缓解这种痛苦，显著提升开发效率。具体步骤如下。

背景：本地有一个页面 localhost:8080/home.html, 本机局域网ip 192.178.100.100，手机与电脑位于同一网络内。

  • 手机上开启一个 web 页面, 地址 192.178.100.100:8080/home.html
  • 手机启动开发者模式，如何开启自行百度
  • 手机连接电脑，并在手机上开启 USB 调试
  • 电脑端打开chrome 浏览器，地址栏输入 chrome://inspect；Remote Target 中找到 手机上已开启的那个web页(192.178.100.100:8080/home.html),点击 inspect, 这时候会开启该页面的 devtool 模式，如果 404 则看下是否需要翻墙；
  • 手机端现在和电脑是同步的，可以在电脑 devtool 上进行调试。

什么是跨域，为什么会产生跨域

跨域是浏览器的同源策略引起的，即server to server 的service访问不存在跨域问题。

当一个资源从与该资源本身所在的服务器不同的域或端口请求一个资源时，资源会发起一个跨域 HTTP 请求。

跨域的几种解决方案

通常在业务中需要处理的 xhr 和 fetch 产生异步请求问题。根据请求方式（get or post）不同来采用不同的技术方案

GET请求

get 请求建议统一采用的 jsonp 方案。

jonsp： 前端js 在 window上 注册 callback 函数，然后动态地创建 script 标签插入 html 里，script标签 src除了带上正常的 get请求参数以外，还有把注册的 callbakc 函数名传递给后端，server 同样要对 get 请求进行callback改造，如果收到callback的
参数，则需要把返回数据拼接成 带callback名的数据返回。 注册的callback函数里是处理返回数据的逻辑

POST请求

post 请求根据后端返回数据的方式不同分为两种

白名单

需要身份的验证，例如带cookie, 前端设置 xhr.withCredentials 为 true, 后端根据 content-type 及 header 区别是简单请求还是预检请求。 需要设置一些对应的字段，主要在 access-control-allow-origin 添加白名单地址 ，另外还需要设置 access-control-allow-credentials等一些字段 来配合前端。

iframe嵌套form请求，jump页跳转方式

sdk 性质的跨域 post跨域请求多用到这种解决方案。 动态地创建 iframe,内嵌 form 表单，将 post 请求 url 设置为 form 的 action,
保持 iframe name 和 form 的 target 一致, form表单请求的返回类型为 html, html内包含返回数据， html内自执行脚本跳转到jump页， 前端通过注册message
事件监听 jump页 的相应的message请求。 百度首页的passport 登录就用到了这种技术

这篇文章来源于一个需求：如何统计ajax请求从发起到成功返回所耗费的时间？ 如果在业务中每个请求的发起和结束时进行打点，计算差值的话，那么工作量将非常巨大，如果要求每个项目封装一个通用方法的话，对于我们这种有好几百个项目的部门来说也同样是不可能的，那么就需要对XMLHttpRequest本身进行功能扩展

虽然在一些情况下this指向不发生改变，无需使用apply，但是使用apply也可以得到正确的结果，从参数扩展、方便记忆的角度话可以统一成同一种模式。

通过apply与call实现继承的方式

  • apply() 方法在指定 this 值和参数（参数以数组或类数组对象的形式存在）的情况下调用某个函数.
  • call() 方法在指定this值和若干个指定的参数值的前提下调用某个函数.
  • 注意： apply() 方法接受的是一个包含多个参数的数组，call() 方法接受的是若干个参数的列表
