什么是跨域，为什么会产生跨域

跨域是浏览器的同源策略引起的，即server to server 的service访问不存在跨域问题。

当一个资源从与该资源本身所在的服务器不同的域或端口请求一个资源时，资源会发起一个跨域 HTTP 请求。

跨域的几种解决方案

通常在业务中需要处理的 xhr 和 fetch 产生异步请求问题。根据请求方式（get or post）不同来采用不同的技术方案

GET请求

get 请求建议统一采用的 jsonp 方案。

jonsp： 前端js 在 window上 注册 callback 函数，然后动态地创建 script 标签插入 html 里，script标签 src除了带上正常的 get请求参数以外，还有把注册的 callbakc 函数名传递给后端，server 同样要对 get 请求进行callback改造，如果收到callback的
参数，则需要把返回数据拼接成 带callback名的数据返回。 注册的callback函数里是处理返回数据的逻辑

POST请求

post 请求根据后端返回数据的方式不同分为两种

白名单

需要身份的验证，例如带cookie, 前端设置 xhr.withCredentials 为 true, 后端根据 content-type 及 header 区别是简单请求还是预检请求。 需要设置一些对应的字段，主要在 access-control-allow-origin 添加白名单地址 ，另外还需要设置 access-control-allow-credentials等一些字段 来配合前端。

iframe嵌套form请求，jump页跳转方式

sdk 性质的跨域 post跨域请求多用到这种解决方案。 动态地创建 iframe,内嵌 form 表单，将 post 请求 url 设置为 form 的 action,
保持 iframe name 和 form 的 target 一致, form表单请求的返回类型为 html, html内包含返回数据， html内自执行脚本跳转到jump页， 前端通过注册message
事件监听 jump页 的相应的message请求。 百度首页的passport 登录就用到了这种技术

为原生API添加功能，即继承原生API并进行扩展

这篇文章来源于一个需求：如何统计ajax请求从发起到成功返回所耗费的时间？ 如果在业务中每个请求的发起和结束时进行打点，计算差值的话，那么工作量将非常巨大，如果要求每个项目封装一个通用方法的话，对于我们这种有好几百个项目的部门来说也同样是不可能的，那么就需要对XMLHttpRequest本身进行功能扩展

方法总结

step1：

先用一个变量，如a，指向原方法。

step2：

新写一个函数覆盖原方法，增加你想要的扩展功能，并在其中调用step1中的a，并通过apply方法保证a执行的时候有正确的this指向和参数传递，保证原方法的正常使用。

虽然在一些情况下this指向不发生改变，无需使用apply，但是使用apply也可以得到正确的结果，从参数扩展、方便记忆的角度话可以统一成同一种模式。

通过apply与call实现继承的方式

  • apply() 方法在指定 this 值和参数（参数以数组或类数组对象的形式存在）的情况下调用某个函数.
  • call() 方法在指定this值和若干个指定的参数值的前提下调用某个函数.
  • 注意： apply() 方法接受的是一个包含多个参数的数组，call() 方法接受的是若干个参数的列表
